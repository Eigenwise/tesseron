Tesseron session resume v2.9.0

How a refresh keeps the same Tesseron session paired with the agent. Four scenes, one accent color per scene, the technical names stay visible.

The pitch. Before v2.9.0, every browser refresh minted a new session, so the user had to read a new claimCode into the agent each time. Now the Vite plugin owns a Session by sessionId, not by browser WebSocket. The browser tab can detach and reattach as many times as it wants inside the idle TTL window (default 4 hours); the gateway-side bridge never sees the disconnect; the agent stays paired. One claim, many refreshes.
1 First connect - the only time the user types the claim code

The browser opens a WebSocket and sends tesseron/hello. The Vite plugin mints a Session, synthesises the welcome (which carries the claim code), and writes the manifest. The user types the code into Claude; the gateway dials with the bind subprotocol; the bridge is bound.

Out of band, between 5 and 6. The browser UI shows the claim code (e.g. AB3X-7K). The user reads it and types "claim session AB3X-7K" into Claude. This is the only human step; everything else is automatic.

What is now persisted. The Session lives in the Vite plugin's memory keyed by sessionId. The browser holds { sessionId, resumeToken } in localStorage. The manifest sits in ~/.tesseron/instances/<instanceId>.json.

2 Page refresh - the Session survives the dead WebSocket

The browser navigates away. Its WebSocket closes. The plugin detaches the browser WS from the Session but keeps the Session alive for the idle TTL window (default 4 hours). The gateway never sees a close - the bridge stays open.

Why the gateway sees nothing. The gateway's WebSocket connection lives between the gateway and the plugin, not the gateway and the browser. When the browser tab closes, only one half of the bridge dies. The plugin is still alive in the dev server process; the bridge to the gateway is still alive.

Configurable. The 4-hour default matches @tesseron/mcp's resumeTtlMs. Override on the plugin with tesseron({ sessionIdleTtlMs: ... }); 0 disables cross-refresh resume entirely.

3 Reconnect - resume, no claim code

The page mounts again, a new WebSocket opens, the SDK sends tesseron/resume with the stored credentials. The plugin looks up the Session by sessionId, constant-time compares the token, rotates it, and synthesises a resume response without a new claim code. The agent never noticed.

What the agent saw. Nothing. Its tool list never changed; no tools/list_changed notification fired. As far as the MCP side is concerned, the same session has been continuously claimed since Scene 1.

Why rotate the token. Resume tokens are single-shot. A second resume with the same token would fail. This protects against token replay if the previous token leaked somewhere (e.g. shared between two tabs).

! Failure paths - the safety net

Two ways the resume path can not happen: the idle TTL elapses without a reattach, or the SDK comes back with a stale token. Both fall through cleanly to a fresh tesseron/hello - the user just sees a new claim code, same as Scene 1.

Path A is bounded. A Session can not "leak" forever - the idle timer is unconditional. A tab opened, claimed, and abandoned costs the dev server ~one Session worth of memory for at most 4 hours.

Path B is graceful. The SDK on TesseronError(ResumeFailed) clears the stored creds and retries with tesseron/hello. The user just sees a new claim code, exactly as on a brand-new tab. No error UI required.

Constant-time compare. Token validation uses constantTimeEqual on equal-length strings, so a brute-force attacker can not learn the right prefix from timing the rejection.

What changed in v2.9.0

  • Vite plugin owns a Session by sessionId; browser WS is just an attachment.
  • Refresh detaches and reattaches; bridge to the gateway stays open across both.
  • Idle TTL sessionIdleTtlMs bounds the hold time. Default 4h.
  • Gateway-mint path also bumped: resumeTtlMs 90s -> 4h.
  • Web SDK auto-persists in localStorage by default.
  • React, Vue, Svelte hooks default resume: true.

What you need to do

  • Bump @tesseron/* to 2.9.0 in lockstep.
  • Reinstall + rebuild. No code changes required.
  • Opt out per-app with { resume: false } for incognito-style flows.
  • Tune the dev TTL with tesseron({ sessionIdleTtlMs: ... }).
  • Verify: claim, refresh, observe same session + rotated token + no new claim code.

When resume does NOT happen

  • Tab idle for more than sessionIdleTtlMs (4h default).
  • Dev server (Vite) restarted - Session memory is gone.
  • Gateway / Claude Code restarted.
  • localStorage was cleared (incognito, dev tools, another tab).
  • App caller passed resume: false explicitly.
  • All of the above fall back to a clean tesseron/hello - same UX as first connect.
